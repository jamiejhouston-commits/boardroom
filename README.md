# Hermes Mobile

Hermes Mobile is a native SwiftUI iOS surface for Hermes Agent. The target is not a Telegram-style wrapper; it should become its own first-class Hermes client with the same sessions, profiles, skills, memory, cron jobs, gateway channels, and management surfaces as Hermes Desktop.

## Current Slice

- iPhone-first SwiftUI app scaffold generated by XcodeGen.
- Chat-first Command Center.
- War Room for agent activity, task lanes, tool calls, memory writes, and delegation.
- Agent profiles with editable `soul.md` files persisted under app support storage.
- Cron, Gateway, Skills, Plugins, MCP servers, Providers, Runtime, and Sandbox management surfaces.
- Mac Relay connection flow for using the real Hermes install on this Mac from the iPhone app.
- Unit tests for seeded profiles and `soul.md` persistence.

## Easiest Local Setup

For the first client-friendly version, the iPhone app should not ask users to edit YAML or create a Telegram bot. The easiest flow is:

1. Install and configure Hermes on a Mac.
2. Run one local relay command on that Mac.
3. Enter the displayed URL and token in the iPhone app.
4. Chat from the phone; the Mac relay calls the real Hermes agent.

Prototype command:

```sh
python3 Scripts/hermes_mobile_relay.py
```

The relay prints something like:

```text
URL:   http://192.168.1.20:8787
Token: <generated-token>
Pair:  http://192.168.1.20:8787/pair
```

On iPhone:

1. Open Hermes Mobile.
2. Go to Gateway.
3. Open Mac Relay.
4. Tap Scan Pairing Code and scan the QR on the Mac pairing page.
5. If QR image support is not installed on the Mac, use the manual URL/token fields.

The relay automatically uses Hermes' own Python virtualenv when available so the pairing page can render a QR image without separate setup.

Keep the relay terminal open while using the app. For a product release, this should become an official Hermes command such as `hermes mobile start`, with Bonjour discovery and QR pairing.

## Runtime Direction

Hermes Desktop is a native shell around the same Hermes Agent core used by the CLI, TUI, dashboard, and gateway. The iOS app should follow that same principle:

1. Embedded mode: bundle or bind an iOS-compatible Hermes core runtime inside the app.
2. Gateway mode: connect directly to `hermes dashboard` / Hermes gateway APIs so the phone app becomes its own messaging surface.
3. Desktop relay mode: pair with a desktop or server host for terminal, filesystem, Docker, Singularity, and other backends iOS cannot host locally.

iOS cannot run the desktop Electron shell or spawn arbitrary CLI subprocesses like a Mac app. Heavy execution backends need either an embedded library/runtime that is shipped with the app or a remote Hermes host.

## Commands

```sh
xcodegen generate
xcodebuild -project HermesMobile.xcodeproj -scheme HermesMobile -destination 'platform=iOS Simulator,name=iPhone 17' build
xcodebuild -project HermesMobile.xcodeproj -scheme HermesMobile -destination 'platform=iOS Simulator,name=iPhone 17' build-for-testing
```
